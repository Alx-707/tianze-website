ASSESSMENT CRITERIA
================================================================================

An item is a RETIREMENT CANDIDATE if ANY of the following hold:
  RC-1  The item has explicit, machine-readable exit criteria (EXCEPTION-CONTRACT)
        AND there is at least one credible signal those criteria may now be met.
  RC-2  The upstream (OpenNext / Cloudflare / Next.js) now handles the need
        natively, with documented evidence (not inference).
  RC-3  The item is documented as legacy/backup in code AND is superseded by
        a currently active replacement in this repo.

An item is a NON-CANDIDATE (MUST RETAIN) if:
  NC-1  It is currently on the critical build/deploy path with no replacement.
  NC-2  Removing it would break a CI check or deployment contract.
  NC-3  It is a REPO-SPECIFIC-CONFIG with no official equivalent that could
        replace it (i.e., the config IS the feature).
  NC-4  It has exception contract exit criteria that are explicitly NOT yet met
        (per code comments or AGENTS.md).
  NC-5  Its retirement requires external upstream evidence not yet available.

An item is DEFERRED if:
  D-1   The retirement question requires a build/smoke run to answer
        (forbidden in this inventory task).
  D-2   The item has unknown/undocumented interaction with other components.
  D-3   Official upstream parity is plausible but not confirmed by docs or code.

────────────────────────────────────────────────────────────────────────────────
Reason: D-2 + D-1 — interaction between the two config layers is undocumented.
  The question "does OpenNext's minify:false override wrangler.jsonc minify:true
  or vice versa?" requires either upstream documentation or a build experiment
  (forbidden in this task).
Recommendation: Clarify the precedence in open-next.config.ts comment or
  align wrangler.jsonc to minify:false to remove ambiguity.

────────────────────────────────────────────────────────────────────────────────
DEFERRED-3: OPEN_NEXT_DEPLOY_ENV naming discrepancy
────────────────────────────────────────────────────────────────────────────────
Reason: D-2 — server-actions-secret-contract.mjs exports OPEN_NEXT_DEPLOY_ENV
  (the exported constant name) whose value is "OPEN_NEXT_DEPLOY" (the actual
  env var name). deploy-phase6.mjs reads process.env.OPEN_NEXT_DEPLOY directly
  and does not import the contract constant. The behavior is correct today, but
  if someone renames the env var in the contract without updating deploy-phase6.mjs,
  the discrepancy would cause a silent failure.
Recommendation: Either have deploy-phase6.mjs import OPEN_NEXT_DEPLOY_ENV from
  the contract, or rename the contract export to OPEN_NEXT_DEPLOY_VAR_NAME to
  clarify it is the env var name, not the env var itself.

────────────────────────────────────────────────────────────────────────────────
DEFERRED-4: PHASE6_ALIAS drift — no dedicated contract checker
────────────────────────────────────────────────────────────────────────────────
Reason: D-2 — PHASE6_ALIAS is declared in alias-shim-exception-contract.mjs
  and injected by build-phase6-workers.mjs, but check-alias-shim-contract.mjs
  only checks SINGLE_WORKER_ALIAS against wrangler.jsonc. There is no enforcer
  that verifies PHASE6_ALIAS appears correctly in generated phase6 worker configs.
  If build-phase6-workers.mjs stops injecting it (analogous to the §24 drift
  incident), no contract check would catch it until deploy smoke fails.
Recommendation: Add PHASE6_ALIAS to the contract enforcement chain, either by
  extending check-alias-shim-contract.mjs to also validate a generated phase6
  config, or by adding a separate check.
