# Codex Dispatch Protocol

Superpowers 工作流中 Codex (GPT-5.4) 参与的决策规则。
仅适用于 BDD (Superpowers) 工作流。Impeccable (设计/dwf) 工作流不派发 Codex。

## Integration Points

三个环节评估 Codex 参与：

### 1. Plan Review — 计划审查

**时机**：`writing-plans` 产出计划文档后、执行前
**命令**：`/codex:rescue` 传入 specs + plans 文件内容
**方式**：3-5 轮对抗，`--resume` 续接同一线程

审查聚焦：任务拆分是否覆盖 spec 需求、步骤是否具体（无占位符）、依赖顺序、边界情况。

**跳过条件**：计划 ≤2 个任务且预估 <30 分钟。

### 2. Task Execution — 任务执行

**时机**：计划审查达成一致、进入 `subagent-driven-development` 逐任务执行阶段
**命令**：`/codex:rescue --write` 传入单个 Task 完整内容
**方式**：每个 dispatch 独立 Codex session，完成后 Claude 验收

见下方「自主派发判断」。

### 3. Code Review — 代码审查

**时机**：所有任务完成后、`finishing-a-development-branch` 前
**命令**：
- 实质性变更（≥3 文件或 ≥100 行）→ `/codex:adversarial-review --wait`
- 小幅变更 → `/codex:review --wait`

**跳过条件**：单文件修复、配置调整、纯文案变更。

## 自主派发判断

逐 Task 评估，同一计划中可以部分 Task 派发、部分自己执行。

### 派发信号（越多越倾向派发）

| # | 信号 | 判断依据 |
|---|------|---------|
| D1 | 规格完整 | 计划中有完整代码块、确切文件路径、预期测试输出 |
| D2 | 独立性 | 不依赖其他正在进行的 Task 的运行时输出 |
| D3 | 机械性 | 写已知模式的代码：CRUD、组件模板、数据迁移、批量文件生成 |
| D4 | 规模 | ≥3 个文件或 ≥100 行新代码 |
| D5 | 验证清晰 | 有二元成功标准：特定测试通过、类型检查通过 |
| D6 | 新建为主 | 主要创建新文件/函数，非深度修改现有复杂逻辑 |
| D7 | 架构复杂度 | 涉及复杂逻辑设计或架构决策，受益于不同模型的独立视角 |

### 保留信号（任一命中即保留）

| # | 信号 | 判断依据 |
|---|------|---------|
| K1 | 设计判断 | 涉及视觉布局、UX 流程、品牌语气、审美决策 |
| K2 | 上下文依赖 | 需要头脑风暴中未完全落入计划文档的隐含决策 |
| K3 | 用户交互 | 实施过程中需要与用户反复确认方向 |
| K4 | 敏感路径 | 修改认证、安全、支付、数据完整性相关代码 |
| K5 | 小范围 | 预估 <10 分钟，派发开销不划算 |
| K6 | 深度修改 | 大量改动现有代码内部逻辑，触点多、理解成本高 |
| K7 | 设计工作流 | /dwf 或 Impeccable skills 产出的任何任务 |

### 决策规则

1. **任一 K 信号命中** → 自己执行，不再评估 D 信号
2. 无 K 信号，**D 信号 ≥3** → 建议派发
3. 无 K 信号，**D 信号 = 2** → 自行判断，倾向自己执行
4. **D 信号 ≤1** → 自己执行

建议派发时告知用户（按场景选择措辞）：
> 机械性任务："Task N 定义清晰且偏机械，我打算交给 Codex 执行，我来验收结果。"
> 架构性任务："Task N 涉及复杂架构/逻辑，我想让 Codex 从不同角度做一版，我来审查。"

用户可接受或否决。

## 用户显式指令

| 用户说 | 行为 |
|--------|------|
| "交给codex" / "codex执行" / "让codex做" | 派发当前/指定 Task |
| "你来做" / "自己执行" / "不用codex" | 自己执行 |
| "全部交给codex" | 逐 Task 依次派发，每完成一个验收再派下一个 |
| 无指令 | 按自主判断标准决定 |

用户显式指令覆盖自主判断，包括否决保留信号。

## 派发执行细节

### 任务执行：传递内容

```
/codex:rescue --write "
执行以下任务：

## 项目上下文
- {从 CLAUDE.md 摘取 1-2 句项目描述}
- 技术栈：{Stack 段}

## 任务内容
{从 plans/ 完整复制 Task N 的全部内容：文件路径、代码块、测试命令}

## 验收标准
- {具体测试命令和预期输出}
- 类型检查通过：pnpm type-check
"
```

不传递头脑风暴上下文、其他 Task 内容、用户偏好。计划文档应自包含。

### 任务执行：结果验收

1. 读 Codex 输出，确认报告状态
2. 运行验收命令（测试、类型检查、lint）
3. 检查代码变更是否符合 spec
4. 通过 → 标记 Task 完成，继续下一个
5. 不通过 → 先尝试自己修复；修不了 → `/codex:rescue --resume`

### 计划审查：传递内容

```
/codex:rescue "
审查实施计划，找出缺陷和遗漏。

## 设计规格
{粘贴 specs/ 文件内容}

## 实施计划
{粘贴 plans/ 文件内容}

聚焦：
1. 任务拆分是否覆盖 spec 所有需求
2. 步骤是否具体可执行
3. 依赖和顺序是否合理
4. 边界情况覆盖
"
```

后续轮次 `/codex:rescue --resume "针对你第 N 点反馈，..."`。
