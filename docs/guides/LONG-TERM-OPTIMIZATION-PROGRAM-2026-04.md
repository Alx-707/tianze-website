# 长期优化系列任务总规划

## 文档目的

这份文档不是补丁清单，也不是“先修几个顺手的小问题”。

它回答的是更长期的决策问题：

- 这个仓库未来 6-12 个月最值得持续投入的优化方向是什么
- 哪些优秀开源经验应该真正引入
- 哪些现在看起来高级，但不该急着搬
- 先后顺序怎么排，才不会越做越乱

一句话说：

**这是一份“长期结构优化总方案”，不是短期修修补补任务单。**

---

## 一句话结论

当前仓库最优策略不是继续堆补丁，也不是立刻重拆成 `next-forge` 那类完整重结构。

最优策略是：

1. 先把**真相源、边界、资产、证明链**继续收口
2. 再把当前仓库打磨成**高标准单站基线**
3. 让未来类似项目主要通过**站点身份层 + 页面表达层 + 公共静态 SEO 层**来替换，而不是先动底层 runtime / proof
4. 只有以后真的出现实际共享需求或结构分化证据，再重新评估更重的结构升级

可以把它理解为：

- **现在先做“长期可演进的高标准单站基线”**
- **不是现在就做“为了未来想象而预付大重构成本”**

## 2026-04-21 执行锁定

当前执行锁定如下：

- 不做共享层
- 不做多站点运行时
- 不做 `src/sites/**` 现行落地
- 不做完整多应用重结构
- 专心把当前仓库做成“真实业务单站 + 可二次开发模板基线”

因此，这份文档里所有“共享底座 / 未来多站点”的内容，都应被理解为：

- **未来评估话题**
- 不是当前实施主线

## 当前仓库的长期判断

## 1. 当前项目已经有的长期价值

当前仓库已经具备不少真正值得长期保留的东西：

- 比常见 starter 更重的 Cloudflare 证明边界
- 比普通官网更成熟的 i18n 运行时约束
- 更完整的表单 / 防滥用 / 幂等保护链
- 更系统的质量护栏与真相检查
- 已经开始形成“单站真相源”意识

这意味着：

**你现在不是从“混乱 demo”起步，而是在一个已有底座上继续做长期治理。**

## 2. 当前仓库的长期风险

真正需要长期处理的，不是“页面不好看”，而是下面四类结构风险：

### A. 真相源还没有彻底压平

虽然已经往 `src/config/**` 收口，但仍存在：

- 兼容包装层偏多
- 文档与代码偶尔漂移
- 页面、消息、内容、SEO、站点身份之间还没完全一眼看懂

### B. 横切能力已经很多，但边界表达还不够统一

当前跨很多模块生效的能力已经不少：

- i18n
- SEO
- Cloudflare
- security
- observability
- proof / quality gate

它们本身不弱，但后续如果不继续做边界整理，就会越来越难维护。

### C. 内容和 SEO 已经存在，但还不够“资产化”

现在不是没有内容体系，而是：

- 内容资产
- SEO 资产
- 站点资产
- 路由资产

这几层还没完全形成一个长期清楚的资产模型。

### D. 未来多站点方向已存在，但升级触发条件必须继续写死

仓库已经很明确：

- 当前不该急着走完整多应用重结构
- 未来如果真的进入多站点深分化，再升级

但这条规则必须继续保持清楚，不能后面又被“模板焦虑”带偏。

---

## 长期优化的总原则

后续所有优化，默认遵守这 6 条。

## 原则 1：先收口真相，再谈共享

不要在真相还分散时就急着抽象。

顺序必须是：

1. 先确定哪个文件说了算
2. 再判断哪些能力只是当前单站基线的一部分
3. 再考虑要不要抽目录或抽包

## 原则 2：先做边界清楚，不先做结构好看

长期可维护，不等于目录看起来高级。

真正重要的是：

- 改品牌信息时，不要碰一堆页面
- 改 Cloudflare 证明逻辑时，不要误伤业务路径
- 改 i18n 时，不要重新把消息真相搞散

## 原则 3：不为了“未来可能”提前承担重复杂度

没有真实分化证据前：

- 不急着拆 `apps/* + packages/*`
- 不急着引入完整多应用壳层
- 不急着把当前仓库改造成 SaaS 平台骨架

## 原则 4：平台边界和业务边界必须继续分开

尤其是 Cloudflare：

- 平台适配问题
- 生成产物兼容问题
- 当前站业务回归
- 最终部署问题

必须继续分层。

## 原则 5：内容、SEO、消息都当正式资产，不当零碎文本

长期要避免的，是“页面能跑，但不知道内容到底归谁管”。

## 原则 6：任何长期优化，都要能接回当前 proof 链

长期治理不能绕开现有验证体系。

换句话说：

**长期方案不是另起炉灶，而是让现有 proof 更有层次。**

## 外部 repo 经验：长期应该真正吸收什么

## 1. `weijunext/nextjs-starter`

### 长期最值得吸收的点

- 单站基础信息集中配置
- i18n 入口更直白
- metadata / routing / site info 不要互相藏逻辑

### 长期不该吸收的点

- 它更轻，不代表它的验证链适合作为当前上限
- 不要为了学它，把当前更重的 proof 回退成轻模板

### 在本仓库里的长期角色

**它是“单站真相收口”参考，不是整体架构答案。**

## 2. `opennextjs/opennextjs-cloudflare`

### 长期最值得吸收的点

- 官方示例对照面
- 平台边界表达
- 构建 / 本地信号 / 部署 proof 分层

### 长期不该吸收的点

- 不要用它的轻示例替换当前更严格的验证面

### 在本仓库里的长期角色

**它是“Cloudflare 真相对照源”。**

## 3. `timlrx/tailwind-nextjs-starter-blog`

### 长期最值得吸收的点

- 内容资产意识
- 站点元信息集中管理
- blog / 内容页 metadata 更系统

### 长期不该吸收的点

- 不要把博客模板整体照搬成官网结构

### 在本仓库里的长期角色

**它是“内容与 SEO 资产治理”参考。**

## 4. `ixartz/Next-js-Boilerplate`

### 长期最值得吸收的点

- 质量门槛分层表达
- CI job 职责分开
- i18n / static / e2e / visual 各自边界清楚

### 长期不该吸收的点

- 不要继续堆更多工具
- 要学体系化表达，不学工具数量

### 在本仓库里的长期角色

**它是“质量护栏分层表达”参考。**

## 5. `vercel/next-forge`

### 长期最值得吸收的点

- 横切能力按主题分层
- `seo` / `internationalization` / `security` / `observability` 的边界意识
- 未来升级窗口规则
- 单一出口能力，例如 `JsonLd` 这种共享出口

### 长期不该吸收的点

- 现在就拆完整多应用结构
- 现在就照着引入过重的 SaaS 风格横切体系

### 在本仓库里的长期角色

**它是“未来结构升级参考图”，不是当前直接落地模板。**

## 长期优化分期规划

下面这 6 期，按顺序推进。

## Phase 0：统一长期话语体系

### 目标

让后续所有长期优化，都建立在一套稳定词汇和边界上。

### 这期要解决什么

- 什么叫共享层
- 什么叫业务真相
- 什么叫站点资产
- 什么叫 proof boundary
- 什么叫升级窗口

### 当前已有基础

- `CANONICAL-TRUTH-REGISTRY.md`
- `QUALITY-PROOF-LEVELS.md`
- `RELEASE-PROOF-RUNBOOK.md`
- `future-upgrade-window-rules.md`

### 下一步要补什么

- 再做一份更适合 owner / AI / 新维护者共同使用的长期入口图
- 把“当前主真相层”和“未来候选共享层”画清楚

### 完成标准

- 后续讨论不再反复回到“哪个文件说了算”
- 新任务能直接落到固定边界

---

## Phase 1：继续收口单站真相层

### 目标

把当前单站模式做成一个长期稳定的高标准基线。

### 这期的核心不是“再修一点”

而是把下面几类东西进一步收口：

- 站点身份
- 页面表达输入
- 公共静态 SEO 输入
- 联系方式
- 公司事实
- 默认 SEO
- 产品目录
- 首页入口承担的业务真相

### 重点吸收来源

- `weijunext/nextjs-starter`

### 应做事项

1. 继续把站点身份层压回 `src/config/single-site.ts`
2. 清理仍然在页面里偷藏业务真相的地方
3. 明确哪些包装层只是兼容层，不允许再新增业务真相
4. 对外形成一页“单站主真相入口图”

### 完成标准

- 改一个站点基础事实，不再需要追多个包装入口
- 页面更多只是消费，不再偷偷定义业务身份

---

## Phase 2：把内容 / SEO / 消息正式资产化

### 目标

把“内容相关的东西”从零散文本提升为长期资产模型。

### 这期要解决什么

- 哪些属于内容资产
- 哪些属于 SEO 资产
- 哪些属于消息资产
- 它们之间的依赖关系是什么

### 重点吸收来源

- `timlrx/tailwind-nextjs-starter-blog`
- `weijunext/nextjs-starter`

### 应做事项

1. 完成内容资产 ledger（账本）
2. 完成 SEO 资产 ledger
3. 把页面运行时真相和 MDX / content 资产边界继续写死
4. 继续收口 structured data 出口
5. 明确 sitemap / metadata / content 的变更影响面

### 完成标准

- 内容、SEO、消息不再是三套平行世界
- 页面内容调整时，能知道影响的是哪类资产

---

## Phase 3：把 proof 链升级成“站点感知 + 长期可扩展”的体系

### 目标

保留当前重 proof 优势，但让它适合未来多站点。

### 这期要解决什么

当前 proof 很强，但仍偏“当前单站高标准”。

未来要补的是：

- 共享层改动时要验证什么
- 站点层改动时要验证什么
- Cloudflare proof 怎么按站点扩
- 内容 / SEO / i18n proof 怎么按站点扩

### 重点吸收来源

- `opennextjs/opennextjs-cloudflare`
- `ixartz/Next-js-Boilerplate`

### 应做事项

1. 把 quality gate 分成更清楚的层级图
2. 把 Cloudflare 官方对照 lane 固化进升级流程
3. 把 site-aware proof 规则从草案推进到执行规则
4. 区分：
   - fast local
   - local-full
   - ci-proof
   - release-proof
   - future site-aware proof

### 完成标准

- 后续不是“又多一堆脚本”
- 而是“每类变更该过哪层 proof”一眼能看懂

---

## Phase 4：开始抽出长期稳定的横切能力层

### 目标

在不重拆仓库的前提下，让横切能力更稳定，并继续服务单站基线模板化。

### 这里说的横切能力，指的是

- SEO
- i18n
- security
- observability
- site identity consumption

### 重点吸收来源

- `vercel/next-forge`

### 应做事项

1. 把共享出口继续统一，例如：
   - metadata helper
   - JsonLd / structured data
   - i18n loader
   - security header / client identity helper
2. 明确哪些能力已足够稳定，可以视作“准共享层”
3. 为未来可能的 `packages/*` 抽离先做逻辑边界，不先做目录重构

### 完成标准

- 同类能力不再多个出口并存
- 即使还在同一仓库目录里，边界也已经更像长期共享层

---

## Phase 5：做“二次开发准备”，不是“多站点重构”

### 目标

把未来多站点真正需要的准备动作做完，但继续避免提前重构。

### 这期要解决什么

- 哪些东西未来类似项目一定会替换
- 哪些 proof 在替换后仍必须保持
- 哪些表达层差异不该继续塞回页面内部

### 重点吸收来源

- `next-forge`
- `opennextjs/opennextjs-cloudflare`

### 应做事项

1. 维护站点差异候选清单
2. 维护 derivative-project replacement checklist
3. 定义模板就绪 proof 的最低组合
4. 把未来二次开发优先替换面做扎实

### 完成标准

- 当类似项目真的开始二次开发时，不需要从零讨论
- 但也没有提前背上完整重结构维护成本

---

## Phase 6：只有触发条件成立后，才进入结构升级

### 目标

把结构升级从“感觉要不要拆”变成“条件满足才执行”。

### 触发条件示例

- 第二、第三站长期并行存在
- 差异不再只是内容和设计
- 发布节奏按站点明显分开
- 站点专属依赖越来越多
- 当前目录下的站点特例已经明显膨胀

### 这时才考虑什么

- 是否升级到更像 `next-forge` 的多应用结构
- 是否抽出更正式的共享包
- 是否按站点拆应用入口

### 完成标准

- 升级是被证据推动，不是被模板推动

---

## 近期 3 个最值的长期动作

如果只看“现在开始推进，但面向长期”，优先做下面 3 件事。

## 1. 做长期总入口，不再靠零散 checklist 驱动

也就是把：

- 真相
- 内容
- SEO
- proof
- 升级窗口

这些长期议题统一到一个程序化入口里。

### 价值

这是后续所有长期工作的总路由。

## 2. 把“单站高标准基线”做扎实

先把当前单站模式做到足够清楚、足够稳定、足够易解释。

### 价值

以后无论是否多站点，这一步都不会浪费。

## 3. 把 proof 体系变成未来可扩展的 site-aware 模型

不是现在就多站点，而是先让规则具备未来扩展能力。

### 价值

以后少挨打，尤其是 Cloudflare 和站点身份这两条线。

---

## 明确不建议现在做的事

## 1. 不建议现在直接上完整多应用壳层

原因：

- 真实业务分化证据还不够
- 当前收益小于结构成本

## 2. 不建议继续靠补丁式局部收拾来假装长期治理

原因：

- 补丁会继续堆高理解成本
- 只会延后结构问题，不会消掉

## 3. 不建议为了“学开源项目”去回退现有更强 proof

原因：

- 当前仓库真正值钱的，就是 proof 比普通 starter 更重

## 4. 不建议现在把 `next-forge` 当成唯一标准答案

原因：

- 它更像未来边界图，不是当前最优落地形态

---

## 这个长期规划之后怎么用

后续默认按这个顺序工作：

1. 先判断当前任务属于哪一相
2. 再判断参考哪个 repo 的经验
3. 再判断是：
   - 先做文档 / 规则收口
   - 还是可以直接做低风险结构落地
4. 最后再决定验证面

换句话说：

**以后不再问“这个想法听起来高级吗”，而是问“它属于哪一相、对应哪类长期目标、会不会提前背上不该背的重量”。**

---

## 当前建议的默认执行顺序

短版如下：

1. 统一长期入口与边界语言
2. 继续压平单站真相层
3. 资产化内容 / SEO / 消息
4. proof 体系升级为未来 site-aware 模型
5. 再抽稳定横切能力层
6. 最后才判断是否触发结构升级

一句话总结：

**先把高标准单站做成长期底座，再决定要不要长成更重的多站点结构。**
