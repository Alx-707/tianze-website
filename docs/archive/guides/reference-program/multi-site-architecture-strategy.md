# 多业务网站架构方案

## 文档目的

这份文档用于回答一个具体决策问题：

在当前项目基础上，同时开发多个业务网站，并且让后续更新和优化尽可能同步时，当下最优的架构方案是什么。

本文不是泛泛讨论常见架构，而是基于当前仓库的实际状态做判断，并给出可执行的落地路径。

## 结论摘要

当下最优方案不是直接上完整的多应用 `monorepo`。

当下最优方案是：

- 保持一个代码仓库
- 保持一套共享网站底座
- 为每个业务拆出独立的站点配置包
- 每个业务网站单独部署、单独绑定域名

可以把它理解为：

- 现在先做“单仓库，多站点配置，多站独立部署”
- 不是现在就做“多应用 monorepo”
- 未来如果站点之间明显分化，再升级为完整 `apps/* + packages/*` 结构

一句话总结：

当前最优解是“共享引擎 + 多站点配置包”，而不是“立即拆成多个独立应用”。

## 当前执行建议

这份文档描述的方向仍然成立，但当前执行建议需要分两种情况。

### 情况一：第二、第三个业务站还没有确认

- 先保留这份架构判断
- 先不启动第一阶段代码拆分
- 先用参考仓库对标清单指导当前项目的日常改造

### 情况二：第二、第三个业务站已经确认

- 正式进入整改阶段
- 先按“站点身份层 -> 内容资产层 -> 页面松耦合 -> 多站点验证”顺序推进
- 暂时仍然不直接升级到完整多应用 `monorepo`
- 默认执行入口改为：
  - [multi-site-real-remediation-checklist.md](/Users/Data/Warehouse/Pipe/tianze-website/docs/archive/guides/reference-program/multi-site-real-remediation-checklist.md)

也就是说，即使业务站已经确认，当前最优动作仍然不是“现在就把仓库结构拆开”，而是先做一轮有顺序的真实整改。

只有先把这一步做好，后面不管是继续维持单仓多站点，还是升级到完整 monorepo，成本都会更低。

## 业务需求拆解

根据当前需求，可以归纳出以下真实诉求。

### 1. 网站骨架高相似

用户明确表示多个网站“结构和当前项目基本一样”。

这意味着：

- 页面模型大概率相似
- 组件复用价值很高
- 多站之间共享更新的价值很高

### 2. 差异主要在内容和设计

用户明确表示“只是内容和设计不同”。

这意味着：

- 当前阶段的主要差异，不在底层逻辑
- 真正要隔离的是站点资料、品牌、视觉和内容
- 不是立即拆成多个独立应用

### 3. 更新和优化需要同步

这是最关键的一点。

这意味着：

- 不能简单复制多个项目长期维护
- 共享底座必须是单一真相源
- 一次修复、多个网站受益，是核心目标

### 4. 多业务同时推进

这意味着：

- 每个业务网站仍然需要独立管理内容和配置
- 不能所有业务强行混在同一份站点配置里
- 部署与域名层面需要站点隔离

### 5. 当前阶段尚未证明站点会快速分化为完全不同产品

从现有描述看，没有证据表明这些网站会在短期内演化成：

- 完全不同的信息架构
- 完全不同的业务流程
- 完全不同的交互模型
- 完全不同的研发团队边界

所以现在就直接拆成多个独立应用，属于提前承担复杂度。

## 评估标准

为了判断“当下最优方案”，这里使用六个维度统一比较。

### A. 同步更新成本

一次优化是否能尽量只改一处。

### B. 当前实施成本

从当前仓库迁移过去，是否会引入过高的重构成本。

### C. 站点独立性

每个业务网站是否能独立配置、独立部署、独立调整内容和视觉。

### D. 长期维护成本

半年到一年后，是否会因为重复代码或结构混乱而拖慢开发。

### E. 对当前项目匹配度

是否贴合“结构类似、内容和设计不同”的现实。

### F. 向未来升级的弹性

如果未来各网站分化加剧，是否容易平滑升级。

## 备选方案对比

## 方案一：复制当前项目，多份仓库分别维护

### 做法

每个业务复制一份完整项目，后续分别修改。

### 优点

- 启动最快
- 每个站完全独立
- 不需要先做架构抽象

### 缺点

- 同步更新成本最高
- 很容易出现改漏和版本漂移
- 后续统一升级最痛苦

### 结论

不适合作为当前长期方案，只适合非常短期的应急复制。

## 方案二：一个应用内用条件判断支持多个站点

### 做法

仍然保留一个应用，根据域名、环境变量或站点标识，切换不同站点内容和主题。

### 优点

- 同步更新天然最简单
- 当前迁移成本最低
- 对现有项目改造最平滑

### 缺点

- 如果站点差异继续增加，很容易变成到处都是条件分支
- 站点之间的边界容易模糊
- 后续可维护性取决于配置边界是否清晰

### 结论

这是当下最接近最优解的方向，但前提是必须把“站点差异”做成独立配置包，而不是散落在各处条件判断里。

## 方案三：完整多应用 monorepo

### 做法

把仓库拆成：

- `apps/site-a`
- `apps/site-b`
- `apps/site-c`
- `packages/ui`
- `packages/core`

### 优点

- 应用边界最清晰
- 每个站点独立性最高
- 适合后续站点持续分化

### 缺点

- 当前阶段实施成本偏高
- 需要更早维护多份应用壳层
- 测试、构建、环境变量、部署配置会明显变复杂
- 在共享度仍然很高的时候，属于过度拆分

### 结论

这是一个好的第二阶段方案，但不是当前阶段最划算的第一步。

## 方案四：多仓库 + 公共底座包

### 做法

每个站点一个独立仓库，共享能力放到单独的公共包里。

### 优点

- 站点独立性高
- 各仓库边界清楚

### 缺点

- 底座升级仍要逐个站点跟进
- 版本协同成本高
- 不适合当前强调“同步更新”的诉求

### 结论

适合未来团队边界非常明确时使用，不是当前最优。

## 方案评估结论

如果只看当前阶段，“结构类似、差异主要在内容和设计、更新要同步”这三个条件同时成立时：

最优选择是方案二的增强版：

- 一个仓库
- 一套共享网站引擎
- 每个业务独立站点配置包
- 每个业务独立部署

它兼顾了：

- 同步更新能力
- 当前实施成本
- 站点独立性
- 未来升级弹性

## 最终推荐方案

## 推荐名称

单仓库多站点配置架构

## 核心思想

共享逻辑只保留一份，站点差异全部进入独立站点配置包。

### 共享层保留内容

- 页面模块
- 表单流程
- SEO 引擎
- 语言与消息加载能力
- 安全、限流、缓存逻辑
- 通用 UI 基础件
- 部署脚本模板

### 站点层独立内容

- 品牌名
- 域名
- 联系方式
- 社媒地址
- SEO 默认文案
- 颜色、字体、视觉 token
- 导航和页脚
- 产品目录
- 页面模块组合顺序
- 内容文件与素材文件

## 推荐目录方向

建议逐步演进到下面这种结构：

```text
src/sites/
  tianze/
    site.config.ts
    site.facts.ts
    theme.tokens.ts
    navigation.ts
    footer.ts
    product-catalog.ts
    messages/
    content/
    public/

  business-a/
    site.config.ts
    site.facts.ts
    theme.tokens.ts
    navigation.ts
    footer.ts
    product-catalog.ts
    messages/
    content/
    public/

src/
  shared app engine
```

当前第一阶段建议直接落在 `src/sites/`，而不是仓库根目录的 `sites/`。这样可以减少额外路径别名和工具链适配成本，先把站点边界抽清楚，再决定以后是否需要继续外提。

如果后面站点越来越多，也可以再进一步演进成：

```text
apps/
  tianze
  business-a
  business-b

packages/
  core
  ui
  seo
  forms
  i18n
  site-runtime
```

但这一步不建议现在先做。

## 为什么它比完整 monorepo 更适合当前阶段

### 1. 它先解决真正的问题

当前真正的问题不是应用数量，而是站点身份和共享逻辑耦合。

先把站点资料抽离，收益立刻可见。

### 2. 它不提前引入过高复杂度

完整 monorepo 会过早引入：

- 多应用管理
- 多套构建入口
- 多套环境变量
- 多套部署壳层

这些复杂度在当前阶段还没有被业务差异真正证明是必要的。

### 3. 它保留了未来升级空间

先做“多站点配置包”不会阻碍未来升级到完整 monorepo。

相反，这一步做好以后，再升级会更顺。

## 演进路线

## 第一阶段：抽离站点配置层

目标：

让当前项目不再把业务信息写死在共享逻辑中。

优先抽离这些内容：

- `SITE_CONFIG`
- `siteFacts`
- `footer links`
- `navigation`
- `product catalog`
- `messages`
- `content`
- 主题样式 token

当前重点来源：

- [src/config/paths/site-config.ts](/Users/Data/Warehouse/Pipe/tianze-website/src/config/paths/site-config.ts)
- [src/config/site-facts.ts](/Users/Data/Warehouse/Pipe/tianze-website/src/config/site-facts.ts)
- [src/config/footer-links.ts](/Users/Data/Warehouse/Pipe/tianze-website/src/config/footer-links.ts)
- [src/lib/navigation.ts](/Users/Data/Warehouse/Pipe/tianze-website/src/lib/navigation.ts)
- [src/constants/product-catalog.ts](/Users/Data/Warehouse/Pipe/tianze-website/src/constants/product-catalog.ts)

## 第二阶段：建立站点选择机制

目标：

让应用可以根据站点标识加载不同站点包。

站点标识可以来自：

- 域名
- 部署环境变量
- 构建参数

建议默认使用构建参数或环境变量，而不是在代码里硬编码判断。

## 第三阶段：落地第二个业务站

目标：

用真实第二个站验证抽象是否合理。

验收重点：

- 是否无需复制共享逻辑
- 是否能独立切换品牌和内容
- 是否能独立部署
- 是否能继续共享表单、SEO、安全和语言能力

## 第四阶段：观察分化程度

观察周期建议为 1 到 3 个版本周期。

如果发现以下情况持续出现，再启动完整 monorepo 迁移：

- 站点页面结构明显分化
- 站点功能差异明显扩大
- 站点之间共享比例明显下降
- 不同站点开始由不同人独立维护

## 升级为完整 monorepo 的触发条件

满足以下任意三项时，可以认为进入第二阶段架构升级窗口：

- 各站点超过 20% 到 30% 的页面结构不再相同
- 各站点开始拥有明显不同的业务流程
- 发布节奏开始彼此独立
- 维护者开始按站点分工
- 某站点需要独立引入较多专属依赖
- 多站点配置架构中开始频繁出现复杂特殊分支

这时再升级到完整多应用 monorepo，时机更成熟。

## 部署建议

虽然当前不建议上完整多应用 monorepo，但建议从一开始就坚持：

- 每个业务网站独立部署
- 每个业务网站独立域名
- 每个业务网站独立环境变量
- 每个业务网站独立 Cloudflare 或平台配置

这能避免站点之间在发布和运维层面相互牵连。

## 风险与控制方式

## 风险一：共享层继续掺杂业务信息

### 表现

共享组件和逻辑里继续直接引用某个业务品牌或产品目录。

### 控制方式

明确约束：

- 共享层不能直接写业务名
- 共享层只能依赖站点包提供的数据

## 风险二：站点差异通过 if/else 散落在全项目

### 表现

代码里到处出现“如果是 A 站就这样，如果是 B 站就那样”。

### 控制方式

统一要求：

- 差异优先做成配置
- 差异再大一点做成可插拔模块
- 最后才允许极少量站点特例分支

## 风险三：过早升级到完整 monorepo

### 表现

刚开始站点还很像，就先拆出多个应用，导致维护和部署复杂度大增。

### 控制方式

先跑通第二个站，再决定是否需要应用级拆分。

## 风险四：拖太久不升级

### 表现

站点已经明显分化，但仍强行塞在一个应用结构中。

### 控制方式

按本文定义的触发条件定期复盘，一旦进入升级窗口，就启动 `apps/* + packages/*` 迁移。

## 成功标准

采用本方案后，阶段性成功的标准应当是：

- 新增一个业务站时，不需要复制整份项目
- 共享逻辑改一次，多个站点都能受益
- 每个站点可以独立替换品牌、内容和视觉
- 每个站点可以独立部署和配置环境变量
- 代码中不再散落大量业务写死信息
- 未来如果需要升级成完整 monorepo，不需要推翻已有分层

## 最终决策

基于当前项目状态和业务诉求，最终建议如下：

### 当下最优方案

采用“单仓库 + 共享引擎 + 多站点配置包 + 独立部署”。

### 当前不建议

当前阶段不建议直接上完整多应用 monorepo。

### 未来升级路径

如果站点后续从“内容和设计不同”升级为“结构和流程也明显不同”，再升级为完整 `monorepo`。

## 后续执行建议

推荐按下面顺序执行：

1. 先定义站点配置包结构
2. 先抽离品牌、联系方式、SEO、导航、页脚
3. 再抽离产品目录、消息和内容
4. 用第二个业务站验证
5. 观察 1 到 3 个版本周期后，再决定是否升级为完整 monorepo

到这一步为止，当前项目既能支持多个业务网站并行开发，也不会过早承担完整多应用架构的复杂度。
