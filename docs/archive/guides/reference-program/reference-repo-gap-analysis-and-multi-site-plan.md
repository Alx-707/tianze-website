# 当前仓库对标 5 个参考仓库的差距清单与调整清单

## 文档目的

1. 这 5 个参考仓库分别该学什么，不该学什么。

## 一句话结论

当下最优选择不是立刻照着 `next-forge` 或任何一个模板开始拆仓库。

当下最优选择是：

- 用 5 个参考仓库分工对标
- 先把当前仓库的改造顺序排清楚
- 先补那些不改大架构也能提升稳定性的部分
- 把第一阶段代码拆分暂时延后，但把验证结论和实施注意点先记下来

简单说：

**现在先建立“参考基线 + 调整清单”，而不是直接大动结构。**

## 5 个参考仓库分别该学什么，不该学什么

## 1. `weijunext/nextjs-starter`

### 该学什么

- 单站、多语言、内容型网站的目录组织方式
- 基础站点信息集中配置，而不是散落在页面里
- 路由、内容、页面职责相对清楚的做法
- 比较轻的起步形态，不一上来就把工具链堆很重

### 不该学什么

- 不要把它当成你当前项目的质量上限
- 不要因为它更轻，就把你现在已有的发布证明和 Cloudflare 验证降级
- 不要照着它的目录直接硬套到当前仓库

### 对当前项目的现实意义

它最值得拿来学的，不是“多站点”，而是：

**怎么把一个单站项目先整理成更容易复用的样子。**

### 可直接转成的动作

- 把站点基础信息继续往集中配置收口
- 让内容、页面、配置的边界更清楚
- 减少“页面里偷偷写业务真相”的情况

## 2. `opennextjs/opennextjs-cloudflare`

### 该学什么

- Cloudflare 构建、预览、部署相关入口怎么收口
- 哪些问题属于平台适配层，哪些才是业务层问题
- 平台边界尽量写进脚本，不靠人脑记
- 本地验证、预览验证、最终部署验证要分层

### 不该学什么

- 不要拿它的简化示例，反过来削弱你现在仓库里更严格的 smoke 习惯
- 不要把官方示例当成你当前仓库的最终证明标准
- 不要为了贴近示例，回退你现在已有的 Cloudflare 保护措施

### 对当前项目的现实意义

它不是教你把站做得更好看，而是帮你：

**以后少把 Cloudflare 链路问题误判成页面代码问题。**

### 可直接转成的动作

- 继续把 Cloudflare 相关证明边界写死在脚本和文档里
- 让“本地信号”和“最终部署证明”继续分开
- 避免把 `build` 通过误当成 Cloudflare 完全安全

## 3. `vercel/next-forge`

### 该学什么

- 边界意识，也就是哪些东西该放共享层，哪些东西该留在站点或应用层
- 国际化、SEO、安全、日志、功能开关这类横切能力的组织方式
- 共享能力应该成组管理，而不是分散在每个页面、每个应用里
- 如果未来真的进入多应用阶段，什么形态才是更顺的升级方向

### 不该学什么

- 现在不要直接学它的完整多应用重量
- 不要为了“看起来正规”就把整套 SaaS 世界常见依赖搬进来
- 不要在当前站点高度相似时，提前承担多壳层的复杂度

### 对当前项目的现实意义

它最有价值的地方是：

**给你一个未来边界长什么样的参考图。**

也就是说，当前应该学它“怎么划边界”，不是学它“现在就怎么拆应用”。

### 可直接转成的动作

- 给共享能力写清楚真相源
- 继续减少共享层里的业务默认值
- 在文档层先定义未来升级窗口，而不是现在就硬拆

## 4. `timlrx/tailwind-nextjs-starter-blog`

### 该学什么

- 内容模型、前置元数据、SEO 相关资产怎么长期保持整洁
- 内容不是零碎文案，而是结构化资产
- 内容布局和内容管线如何避免越写越乱
- 长期内容维护时，哪些信息应该集中声明

### 不该学什么

- 不要把它的博客思路强行套到当前官网
- 不要完整照搬它的内容工具链
- 不要忽略你当前双语和业务内容的特殊约束

### 对当前项目的现实意义

它在提醒你：

**内容层本身就是一个应该被管理的系统，不只是页面上的几段字。**

### 可直接转成的动作

- 盘一遍当前内容资产有哪些类别
- 先梳理哪些内容应该被结构化，而不是继续散在页面里
- 给后续内容和消息文件的收口预留原则

## 5. `ixartz/Next-js-Boilerplate`

### 该学什么

- 默认质量门槛怎么铺
- 提交流程、检查脚本、测试和开发者体验怎么组合成体系
- 哪些检查适合做成默认规则，而不是靠人记
- 质量护栏如何变成日常流程的一部分

### 不该学什么

- 不要为了“完整”而把数据库、认证、监控、Storybook 等一整套重量都搬进来
- 不要让工具体系反过来主导你的业务结构
- 不要把“工具更多”误当成“质量更高”

### 对当前项目的现实意义

它最适合拿来做：

**查漏补缺清单。**

也就是看看你现在的质量护栏还缺哪几块，而不是把它当作新的项目模板。

### 可直接转成的动作

- 继续收紧哪些检查必须绿灯
- 减少“看起来有脚本，但边界不清”的地方
- 让后续多站点相关验证，也能沿用现在这套护栏思路

## 跨这 5 个仓库提炼出来的调整清单

下面这份清单按“现在就能做”到“先别做”来排。

## A. 现在就能做，而且值得先做

### 1. 先把“共享真相”和“业务真相”写清楚

- 哪些文件是共享底座
- 哪些文件仍然写死了 Tianze 的业务事实
- 哪些内容以后很可能成为多站点差异

这一步不改大结构，但能减少以后重复判断。

### 2. 把参考仓库变成固定对标面

建议直接固定分工：

- 站点组织参考：`weijunext/nextjs-starter`
- Cloudflare 真相参考：`opennextjs/opennextjs-cloudflare`
- 工程边界参考：`vercel/next-forge`
- 内容组织参考：`timlrx/tailwind-nextjs-starter-blog`
- 质量门槛参考：`ixartz/Next-js-Boilerplate`

以后讨论“要不要这么做”时，先看它属于哪一类问题，不要所有问题都指向一个模板。

### 3. 继续把当前仓库的改动决策写成规则，而不是口口相传

优先补这几类规则：

- 哪些东西不能继续写死在共享层
- 哪些验证是发布前必跑的
- 哪些 Cloudflare 问题不能和业务回归混为一谈
- 哪些文档属于真相源

### 4. 先整理“站点差异候选清单”，但不立刻落代码

建议先把未来高概率会变成站点差异的东西列出来：

- 品牌信息
- 联系方式和社媒
- 公司事实
- 产品目录
- 导航和页脚
- 默认 SEO 文案
- 消息文件
- 内容文件
- 主题和视觉 token

先列清单，比现在就开始拆代码更划算。

### 5. 把多站点证明需求先写进检查思路

虽然现在还没开始第一阶段代码拆分，但可以先把要求写清楚：

- 以后如果改共享层，最低证明面是什么
- 以后如果改站点层，最低证明面是什么
- 哪些验证是站点级，哪些验证是共享级

这一步现在写清楚，后面少挨打。

## B. 适合近期继续准备，但暂时不落代码

### 6. 先定未来目录原则，不急着真的搬文件

可以先在文档里约定：

- 未来如果开始第一阶段，推荐把站点差异集中进单独站点层
- 共享层默认禁止继续偷放业务真相
- 只有当配置表达不了时，才考虑更复杂的可插拔逻辑

### 7. 先定义未来升级到 `next-forge` 类结构的触发条件

建议把触发条件写死，而不是到时候凭感觉判断：

- 站点页面结构明显持续分化
- 发布节奏开始按站点分开
- 专属依赖越来越多
- 团队开始按站点而不是按共享层分工
- 当前项目里出现越来越多不好收口的站点特例

只要这些条件还没明显成立，就不应该抢跑到完整多应用结构。

### 8. 为“第二个站点验证”先准备验收问题

以后真的接第二个站时，重点不是“能不能跑起来”，而是：

- 共享层是不是还偷带 Tianze 语境
- 内容和消息是不是容易分开
- 部署和环境是不是能天然隔离
- 新站点是不是必须复制很多现有逻辑

这些问题现在先写出来，后面再做更省事。

## C. 当前明确先别做

### 9. 先不要启动完整第一阶段代码拆分

原因不是方向错，而是时机未到。

当前更划算的是：

- 先把判断标准写清楚
- 先用参考仓库指导日常改造
- 等第二个站点需求更确定时，再开始真正拆第一阶段

### 10. 先不要迁移到完整多应用 `monorepo`

这一步现在太早。

它会先带来：

- 更多壳层维护成本
- 更多构建和环境配置复杂度
- 更多“明明业务还很像，却已经开始分仓治理”的额外负担

### 11. 先不要为了“像成熟模板”而引进更多重量级工具

现在最缺的不是更多工具，而是更清楚的边界和判断。

## 已保存的第一阶段验证记录

虽然本轮决定撤回第一阶段代码落地，但有几条验证结论值得保留，后面真要启动时可以直接复用。

### 1. 如果未来再启动第一阶段，优先考虑轻量站点层，不要先上完整多应用

这条结论没有变。

也就是说，未来真要开始拆，第一步仍然应该是：

- 先把站点差异收口
- 不是先拆成多个独立应用

### 2. 如果未来要试做站点层，优先从这些内容开始

- 站点基础配置
- 公司事实
- 导航
- 页脚
- 产品目录

这批内容最像站点身份，最适合作为第一批抽离对象。

### 3. 一次已撤回的本地试装验证，证明这类轻量抽离不是“天然会炸”

这次试装里，围绕上面那批站点身份信息做过一轮本地验证，曾通过：

- `pnpm type-check`
- 相关针对性测试
- `pnpm build`
- `pnpm build:cf`

这不代表“现在就该做”，但代表以后真要做时，不是完全没有底。

### 4. 如果未来再做本地页面验证，不要把 dev server 发现结果直接当成页面证明

这次试装还暴露了一个值得记住的边界：

- Next 的本地诊断工具能发现 dev server
- 但直接请求本地页面和浏览器自动化仍可能超时

所以以后如果再做这类试装，页面级证明仍要单独确认，不能因为“诊断工具能连上”就宣布页面链路没问题。

## 最终判断

如果只用一句话总结这份文档：

**现在最该做的，不是开始拆第一阶段代码，而是把这 5 个参考仓库变成你的固定对标面，再按这份调整清单持续收紧当前仓库。**

这不是翻盘级动作，但能减少后面继续踩坑的概率。

简单说：

- 现在先把判断标准立住
- 先把该学什么、不该学什么说清楚
- 先把容易做错的地方写进清单
- 等时机到了，再启动第一阶段代码拆分
